Introduction

We need a way for a component to grab another component’s content as its own.

This is required to create dynamic component composition. As an example, a layout component needs to have this because it doesn’t know which components will have to render in its own template. The developer will specify them dynamically when using such layout component.

Currently this is being solved with an additional component, the ComponentPlaceHolder. When designing a template that needs to display one or more unknown components — as it is the case of a layout component — a ComponentPlaceHolder is used instead.

When instantiated the ComponentPlaceHolder grabs and hosts the component that is located at a specific property in its owner component and acts as a proxy to it. This property name is defined on the “name” property of the ComponentPlaceHolder.

When the ComponentPlaceHolder is asked to be drawn in reality it will be the host component that will be drawn at the ComponentPlaceHolder’s location in the DOM tree.

Here’s an example using a two column layout component:

TwoColumnLayout.html

<script type="text/montage-serialization">

{

    "owner": {

        "module": "two-column-layout.reel",

        "name": "TwoColumnLayout",

        "properties": {

            "element": {"#": "two-column-layout"}

        }

    },

   

    "leftPanel": {

        "module": "montage/ui/component-placeholder.reel",

        "name": "ComponentPlaceHolder",

        "properties": {

            "name": "leftPanel",

            "element": {"#": "leftPanel"}

        }

    },

   

    "rightPanel": {

        "module": "montage/ui/component-placeholder.reel",

        "name": "ComponentPlaceHolder",

        "properties": {

            "name": "rightPanel",

            "element": {"#": "rightPanel"}

        }

    }

}

</script>

<body>

    <div id="two-column-layout">

        <div id="leftPanel" style="float: left"></div>

        <div id="rightPanel" style="float: right"></div>

    </div>

</body>

The problem with this approach is the increase of the component tree with proxy components that will have a memory and performance impact in the application depending on how many they are.

The current approach also doesn’t work with any random DOM Node and doesn’t make it possible to grab the entire contents of a component as it is just a simple proxy.

Functional Description

This specification proposes a completely different approach and a generic way to specify the contents of any component. The contents is everything that lies inside the element of a component.

All the components that have elements who are directly connected to a component’s contents will have this component as its parent component. This invariant should be be maintained across changes in a component’s contents, either directly or indirectly.

Here’s an illustration:

+--Component's element

    |

    +--Component A's element

    |

    +--Component B's element

        |

        +--Component B1's element

"Component A" and "Component B"'s element are directly connected to Component's element contents, so their parent component is "Component".

In addition to be able to set a component’s contents we also need to access the original contents of a component. These contents are part of the component configuration and, independently on how the DOM structure of a component changes throughout its life span, they should remain the same and accessible at any time.

With these mechanisms in place it’s easy to set a component’s content out of the original markup of any component or any other user defined elements.

This enable us to grab the entire content of the owner component of a template and set it as the content of a repetition (like in the case of the List).

Another use is to set the content of different components in a template (e.g.: two different Slots) out of well defined properties of the owner component, the elements assigned to these properties are defined by the developer that is using the component.

API

A new Component property — content — that gets and sets the node list of the component’s element, it can be assigned a single Node or an array-like object of Nodes.

A new Component property — originalContent — that holds the user defined markup of a component. This element is non-writable and set the first time a component is assigned an element.

Examples

Here’s how to implement the List using this specification:

"repetition1": {

    "module": "montage/ui/repetition.reel",

    "name": "Repetition",

    "properties": {

        "element": {"#": "repetition"}

    },

    "bindings": {

        "content": {

            "boundObject": {"@": "owner"},

            "boundObjectPropertyPath": "originalContent",

            "oneway": false

        }

    }

Concerns

  1. Not sure if the “content” property can have other uses than the ones ComponentPlaceHolder was meant to solve.
  1. The only way to assign the contents of a component to the original contents of another component is by establishing a binding, it’s not possible in the serialization to access another object’s property. Since the original contents is a property that doesn’t change, this binding will never be used ever again and its only usefulness is to serialize the component if used inside a repetition.
  2. Clashes with the current Slot API.
